## Dependency lock

The base image is pinned by multi-architecture digest. Direct apt inputs are exact-version pins in the Dockerfile. Python build/runtime upgrades are locked by exact version and wheel SHA256 in `locks/requirements-*.lock`, installed with `--require-hashes --no-deps`. DeepGEMM and both vLLM source lineages are commit-pinned. The build therefore fails closed if a named package or wheel disappears or changes rather than accepting a newer dependency. Functional reproducibility is still gated by the no-cache build, module/source hashes, SM 12.1 custom-op smoke, mounted-artifact boot, independent sentinels, greedy behavior, and the measured 5x64 run.

The `cubins/w2` set is byte-for-byte from the pinned public vLLM-Moet commit. `cubins/w3` is the six-file e43-LUT family used by the validated artifact. `receipts/cubins.sha256` is generated over both directories. The W3 generator reproduced every reference SASS file byte-for-byte, and public cubit commit `c139df8b34f1dcab607f8ccb685fdea948f3ae4d` rebuilt all six runtime cubins with matching SHA256 hashes.

The Docker build records the newly built wheel hash at `/opt/repro/build-wheel.sha256`. The cold/no-cache ancestor build produced wheel SHA256 `ce79f01cac2dc62bb4424b8f85dc70a5f35f3b454778537e94aafb25bf67cc38`; it is byte-different from the live reference because the publication-safe source changes only the attribution comment and wheel ZIP metadata/timestamps also differ. That ancestor image (`sha256:a15829cc659e94764132825ff77803f58b2ea9de1c879724b0e4ecde43892ee2`) exposed two real mounted-artifact failures: an orphan `flashinfer_cubin` namespace and excessive checkpoint allocation before the prepacked path was selected.

The shippable source adds the exact FlashInfer cleanup/import assertions and `0003-prepacked-memory-fix.patch`. Its functionally validated image is `sha256:9580f855d49cd783d494117ebbc5a172236cc401a087339d0dcf85d080749c74` (24,019,694,723 bytes), with Dockerfile SHA256 `a0001c5f8c7100ffdbc7e0bb4edfc985f47ae4857c9fda75a9c3fcf44a00328f`. It was rebuilt incrementally from the no-cache ancestor layers, then booted from the immutable read-only artifact and passed greedy32, all three on-path sentinels, and 5x64 serving. The final source was not falsely represented as a second cold/no-cache image build; `receipts/build_receipt.json`, `receipts/docker-build-nocache-a158.log`, and `receipts/container-e2e.json` separate those evidence stages. Exact runtime hashes, CUDA import, real-plane validation, sentinels, greedy output, and measured serving are the functional reproducibility gate.

The original 64-window served-vs-offline receipt is `receipts/quality_seal_public.json`; its unsanitized source SHA256 was `91500e508e9148944b56f8eb18d8f1c490339fd5ae6663ffdb2bec347706b936`. The paired 64-window bitwise control is `receipts/attribution_control_public.json`; its unsanitized source SHA256 is `05d46ac204f32df230357ddfe1d3a1f67b1a6dd7b6080bb69c99621201fa41b0`. The isolated warp-vs-bitwise delta is +0.0002967106% (PASS at 0.3%), while the bitwise served path itself remains +1.3438157095% above offline. The unresolved common serve-path drift is why this package is labeled a research reproducer rather than a product-quality release.
